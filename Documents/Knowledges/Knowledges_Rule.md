# Knowledge Gen 工作规范

所有知识文档需要遵循本文规范


## 一、知识库检索

### UE相关知识 检索策略

1. 可以通过knot 进行知识检索 **并行检索**：同时发起 `data_type=git`（源码片段）和 `data_type=git_iwiki`（架构总结），获取更全面的信息

   - `data_type=git`，`search_domain=UnrealEngine@UnrealEngine-ue5-main`
   - `data_type=git_iwiki`，`search_domain=UnrealEngine@UnrealEngine-ue5-main-git_iwiki`
   - **knowledge_uuid**: `d890d83194b04c8aad24d0e904cdb762`

2. **keyword**：从分析主题中提取核心类名、函数名、模块名

3. **query**：用自然语言描述要查找的内容

### Knot 输出解析

Knot 搜索结果为 JSON 数组，当结果较大时会被截断并保存到文件。

## 二、文档结构

一篇文档的基本骨架是"开头 → 正文 → 收尾"，具体章节编排由主题决定，不要求固定顺序。

### 2.1 文档头部

```markdown
# 中文标题

> **所属模块**: [模块名称] → [子系统/类名]
> **关键源码**: `File1.h`, `File2.cpp`, `File3.h`
```

紧跟 1-3 句概述，直奔主题，说清本文分析什么、为什么值得看。

### 2.2 正文

按主题自然组织 `##` 小节，不限数量和顺序。以下是常见的内容块，按需组合，不必每篇都覆盖所有类别：

**结构与流程类**（讲清楚"有什么、怎么跑"）：

- 整体架构 / 调用链总览图
- 核心类或数据结构分析（带注释的源码 + 字段说明 + 生命周期图）
- 关键函数实现分析（带注释的源码 + 调用图 / 被调用图）
- 生命周期 / 状态机 / 数据流（用 ASCII 步骤图展示完整阶段变迁，见 4.7）

**实现原理类**（讲清楚"为什么这么做、背后的思路"）：

- 设计意图与取舍——为什么选这个方案、放弃了哪些替代方案、换来了什么好处
- 核心算法 / 策略拆解——把关键逻辑用伪代码或分步骤讲透，不只是贴源码
- 内存布局与性能考量——对象怎么排列、为什么用这种容器、缓存友好性、时间/空间复杂度
- 边界情况与防御性设计——空指针、重复调用、线程竞争等场景下代码怎么兜底
- 常见误用与调试技巧——哪些用法容易踩坑、出问题时怎么快速定位

**环境与配置类**（补充上下文）：

- 线程模型与同步机制
- CVar / 配置参数表
- 与其他系统的交互关系
- 版本演进——如果实现在不同引擎版本间有重大变化，值得点出差异
- 等等

### 2.3 小结

文末给出几条关键结论，说清"这个系统最值得记住的点是什么"。

### 2.4 标题下方的简要说明

每个核心类、结构体、函数、流程的 `##` / `###` 标题下方，先用几条 markdown list 给读者一个快速认知，再展开源码和图表。

写什么取决于读者拿到这个标题时最想知道什么——不同类型的主题，关注点不一样。以下是几个参考示例，实际写的时候根据主题灵活调整，不要死套模板：

**类 / 结构体**——读者想知道"这是什么、怎么来的、干什么用"：

```markdown
## FXxxManager

- [模块名] 的核心管理器，负责 [一句话说清职责]
- 由 [谁] 在 [什么时候] 创建，[持有方式] 被 [谁] 持有
- 核心职责：[职责 1]、[职责 2]、[职责 3]
```

**函数**——读者想知道"谁调用它、做了什么、结果给谁"：

```markdown
### DoSomethingCore

- [这个函数完成什么工作]
- 调用方：`ParentClass::Setup()`、`SomeSystem::Initialize()` 等
- 关键步骤：验证输入 → 查找/创建资源 → 注册到全局系统 → 通知依赖方
- 结果去向：[执行完之后谁能感知到变化]
```

**流程 / 算法**——读者想知道"什么时候触发、怎么跑、约束是什么"：

```markdown
## [流程/算法名称]

- [用一句话概括这个流程的目的]
- 触发时机：[什么条件下 / 被谁调用时执行]
- 输入 → 输出：[数据从哪来] → [产出什么结果]
- 约束：[线程要求、锁、时序依赖等，没有就不写]
```

**枚举 / 标志位**——读者想知道"有哪些值、谁写谁读"：

```markdown
### ESomeFlags

- [这组标志位控制什么行为]
- 设置方：[哪些函数 / 流程会写入]
- 读取方：[哪些系统根据标志做判断]
- 关键值：Flag_A / Flag_B / Flag_C ...
```

写的时候把握两点就行：

1. **开头一句话说清"是什么"**，让读者不用看完全部就能判断要不要继续读
2. **后面的条目按需挑选**，什么对理解当前主题最有帮助就写什么——可以是创建时机、调用方、关键步骤、设计动机、常见误区……不必每次都凑满固定数量

## 三、代码块规范

### 3.1 基本格式

所有 UE5 源码使用 ` ```cpp ` 代码块包裹。

### 3.2 文件定位注释

每个代码块开头标注源码路径：

```cpp
// ============================================================================
// 文件: Engine/Source/Runtime/[Module]/Private/[FileName].cpp
// 函数: [ClassName]::[FunctionName]
// 位置: [一句话定位]
// ============================================================================
```

### 3.3 中文内嵌注释

**核心原则**：以码说话，但不是代码堆砌。通过内嵌注释把“做什么 + 为什么这样做 + 数据流向”讲清楚。

注释要求：

- 不允许只贴纯源码；关键分支条件、输入输出和副作用必须解释
- 长代码块按步骤拆分注释，用步骤分隔符划清阶段
- 用 ★ 标记核心调用，让读者一眼看到最关键的行
- **禁止低信息量注释**（如“获取变量”“设置参数”这类纯复述）

注释层级参考：

```
// ==================== 步骤N：标题 ====================     （大阶段分隔）
// ========== 子步骤标题 ==========                          （小步骤分隔）
[代码行];                    // 行内中文注释
[代码行];                    // ★ 标记核心调用
```

示例（好的注释 vs 坏的注释）：

```cpp
// 坏：纯复述名字，没有信息增量
float MinSampleWeight; // 最小样本权重

// 好：解释语义和影响
float MinSampleWeight; // 样本权重下限，低于该值会被抑制，用于降低噪声放大
```

```cpp
// 坏：只说“做了什么”，不说“为什么”
if (bIsVisible) // 如果可见

// 好：解释分支条件的业务含义
if (bIsVisible) // 只有通过视锥剔除+遮挡剔除的光源才进后续阴影计算
```

### 3.4 UPROPERTY 注释

```cpp
/**
 * The list of actions to execute when this game feature is activated/deactivated
 *
 * ── 插件激活/停用时执行的动作列表
 * ── 定义了插件生命周期中的具体行为
 */
UPROPERTY(EditDefaultsOnly, Category = GameFeature)
TArray<TObjectPtr<UGameFeatureAction>> Actions;
```

### 3.5 代码块编号

多个代码块时用 `[1]`, `[2]`, `[3]` 编号：

```cpp
// ============================================================
// [1] 状态机初始化
// ============================================================
```

### 3.6 函数代码展示规范

当文中以“函数实现”方式展示某个函数时：

- **完整优先**：尽量给出完整实现（从签名到右大括号），避免读者看到一半不知道后面还有什么
- **一个函数一个代码块**：同一个函数的代码应放在一个连续的 `cpp` 代码块中，**不要拆成多个代码块分散在不同的 `###` 小标题下**。读者需要看到完整的函数签名包裹住所有逻辑，才能建立"这些代码属于同一个函数"的认知
- **长函数先骨架后展开**：如果函数确实很长，先给一个带省略标注的完整骨架（从签名到右大括号，中间用 `// ... 步骤N 省略 ...` 占位），让读者一眼看到函数的整体结构；然后再逐段展开每个步骤的细节
- **块内用步骤注释划分阶段**：函数内部的不同阶段用 `// ===== 步骤N：标题 =====` 分隔，这样既保持代码连续，又让读者一眼看清函数内部的逻辑分段。示例：

```cpp
UObject* StaticAllocateObject(const UObjectInitParameters& Params)
{
    // ==================== 步骤1：名字处理 ====================
    if (bCreatingCDO)
    {
        InName = InClass->GetDefaultObjectName();
    }
    // ...

    // ==================== 步骤2：分配内存 ====================
    if (Obj == nullptr)
    {
        Obj = (UObject*)GUObjectAllocator.AllocateUObject(TotalSize, Alignment);
    }
    // ...

    // ==================== 步骤3：全局注册 ====================
    // ★ placement new 构造 UObjectBase，触发 AddObject
    new ((void*)Obj) UObjectBase(InClass, InFlags, ...);

    return Obj;
}
```

- **允许有标注的省略**：如果函数过长，可以省略非核心分支，但要标注省略了什么、为什么不重要：

```cpp
    // ... 此处省略了 error handling 和 edge case 分支，
    //     它们不影响主流程理解，完整实现见 XXX.cpp L200-L250
```

- **如果完全不展开**：明确声明“本节不展开实现，仅说明调用关系”
- 核心函数代码块应覆盖：输入约束、关键分支、输出去向，并至少有 1 处 ★ 核心调用标记

## 四、ASCII 图示推荐

### 4.1 基本要求

- **必须** 放在 ` ``` ` 裸代码块中（无语言标记）
- 图放在对应概念首次出现的位置附近

### 4.2 中英文规则（核心）

中文字符是双宽度，在框图中会导致排版错乱。严格遵守：

| 图表类型        | 语言要求                             | 中文说明方式             |
| --------------- | ------------------------------------ | ------------------------ |
| 树形结构图      | 节点名用英文，行尾 `//` 注释可用中文 | 中文写在 `//` 注释中     |
| 框图 / 包围框图 | **全部使用英文**                     | 图后紧跟「术语说明」表格 |
| 分隔符标题图    | 标题用英文                           | 图后紧跟中文段落说明     |
| 流程步骤图      | 树形 `├─` + 步骤编号 + 英文关键词    | 行尾 `//` 注释可用中文   |

### 4.3 树形结构图（类继承、数据结构）

**类成员展示**：节点名英文，行尾 `//` 中文注释。

```
FSceneRenderer                                             // 场景渲染器
├─ [Member] TArray<FViewInfo> Views                        // 视图数组
├─ [Member] TArray<FVisibleLightInfo> VisibleLightInfos    // 全局光源信息
│   ├─ [Index 0] FVisibleLightInfo (Light 0)               // 第一个光源
│   │   ├─ AllProjectedShadows                             // 所有投影阴影
│   │   └─ ShadowsToProject                                // 需要投射的阴影
│   └─ [Index N] FVisibleLightInfo (Light N)
│       └─ ...
└─ [Member] FScene* Scene                                  // 场景指针
```

**数据流向展示**：用树形图表达数据从输入到输出的变换链路，节点用中文括号标注类名，叶子节点列出关键字段并附中文说明。

```
FPrimitiveSceneInfo                                        // 场景图元
└─ Visibility Culling
    └─ FMeshBatchAndRelevance                              // 相关网格批次
        └─ FMeshPassProcessor                              // 网格通道处理器
            └─ FMeshDrawCommand                            // 网格绘制命令
                └─ FVisibleMeshDrawCommand                 // 可见网格绘制命令
                    ├─ MeshDrawCommand: const FMeshDrawCommand*
                    ├─ SortKey                              // 排序键
                    ├─ StateBucketId                        // 状态桶ID(用于实例化合并)
                    ├─ PrimitiveIdInfo                      // 图元ID信息
                    └─ CullingPayload                       // 剔除负载数据
```

### 4.4 函数调用链

用 `★` 标记核心函数，用 `[Phase N]` 标注执行阶段：

```
FDeferredShadingSceneRenderer::Render()                    // 延迟渲染器主入口
└─ BeginInitViews()                                        // 初始化视图
    └─ LaunchVisibilityTasks()                             // 启动可见性任务
        ├─ [Phase 1] Create View Packets
        │   └─ FVisibilityViewPacket()                     // 为每个视图创建任务包
        │
        ├─ [Phase 2] Setup Task Dependencies
        │   ├─ Tasks.FrustumCull.AddPrerequisites(...)     // 视锥剔除依赖
        │   └─ Tasks.OcclusionCull.AddPrerequisites(...)   // 遮挡剔除依赖
        │
        └─ [Phase 3] Launch Frustum Cull
            └─ ★ FrustumCull(...)                          // 【核心函数】视锥剔除
```

### 4.5 分隔符标题图（大阶段划分）

```
==============================================================================
                    Phase 1: Visibility Computation
==============================================================================
[Render Thread]
│
├─ ComputeViewVisibility()                                 // 计算视图可见性
│   ├─ FrustumCull()                                       // 视锥剔除
│   └─ OcclusionCull()                                     // 遮挡剔除
│
└─ ComputeLightVisibility()                                // 计算光源可见性

==============================================================================
                    Phase 2: Shadow Setup
==============================================================================
[Render Thread]
│
├─ InitDynamicShadows()                                    // 初始化动态阴影

### 4.6 数据关系图

```
FSceneRenderer
├── Views[] ────────────────────> FViewInfo

└── Scene ──────────────────────> FScene (Shared)
```

### 4.7 生命周期图

分析类/对象/资源时，用树形步骤图展示从创建到销毁的完整生命周期。每个阶段标注触发函数和关键状态变化：

```
FXxxResource Lifecycle
│
├─ [1] Create                                          // 创建阶段
│   ├─ Trigger: FXxxSystem::AllocateResource()         // 触发函数
│   ├─ State: Uninitialized                            // 初始状态
│   └─ Owner: FXxxSystem (TUniquePtr)                  // 持有方和持有方式
│
├─ [2] Initialize                                      // 初始化阶段
│   ├─ Trigger: FXxxResource::Init()                   // 触发函数
│   ├─ Depends: Config loaded, Subsystem ready         // 前置依赖
│   └─ State: Uninitialized -> Ready                   // 状态变迁
│
├─ [3] Active Use                                      // 活跃使用阶段
│   ├─ Updated by: FXxxSystem::Tick()                  // 谁驱动更新
│   ├─ Consumed by: Renderer, Physics, ...             // 谁消费
│   └─ State: Ready (can be Suspended/Resumed)         // 运行期状态
│
├─ [4] Pending Destroy                                 // 待销毁阶段
│   ├─ Trigger: MarkForDestroy() / GC mark             // 触发条件
│   ├─ ★ BeginDestroy()                                // 释放外部关系
│   └─ State: Ready -> PendingKill                     // 状态变迁
│
└─ [5] Final Destroy                                   // 最终销毁
    ├─ Trigger: GC sweep / IncrementalPurge            // 触发函数
    ├─ ★ FinishDestroy()                               // 最终清理
    └─ State: PendingKill -> Freed                     // 内存释放
```

关键点：

- 每个阶段标注 `Trigger`（谁触发）、`State`（状态变迁）
- 持有方和持有方式在创建阶段标注，如 `TSharedPtr` / `TUniquePtr` / 裸指针
- 如有条件分支（如"可被挂起/恢复"），用子节点展开
- 用 `★` 标记生命周期中最关键的调用

### 4.8 绘制注意事项

1. **绝不在框线内部放置中文** —— 中文双宽度字符会破坏对齐
2. **树形图的中文只出现在 `//` 注释中** —— 不影响缩进对齐
3. **框图可用 `┌─┐` `└─┘` `│` Unicode 框线字符** —— 框内全英文时不影响排版
4. **分隔符标题用 `===`** 划分大阶段，标题居中、使用英文
5. **每张框图后必须附「术语说明」表格**

## 五、CVar 表格（如有）

如果分析的模块涉及控制台变量，用以下格式列出：

| CVar 名称   | 全局变量  | 默认值 | 类型    | 功能描述 |
| ----------- | --------- | ------ | ------- | -------- |
| `r.Xxx.Yyy` | `GXxxYyy` | `1`    | `int32` | 功能说明 |
| `r.Xxx.Zzz` | `GXxxZzz` | `0`    | `int32` | 功能说明 |

同时附上 CVar 定义的源码（带注释）：

```cpp
// ============================================================================
// 控制台变量定义
// ============================================================================
static int32 GXxxYyy = 1;

static FAutoConsoleVariableRef CVarXxxYyy(
    TEXT("r.Xxx.Yyy"),
    GXxxYyy,
    TEXT("English description"),                              // 英文原始描述
    ECVF_RenderThreadSafe
);
// 功能：中文功能说明
// 默认值 1 表示启用，0 表示禁用
```

## 六、写作风格

### 6.1 语言

- 正文、代码注释：**中文**
- 代码、UE5 术语：**英文原文**
- 首次出现的英文术语附中文翻译：如 "FrustumCull（视锥剔除）"
- ASCII 图表：见第四节

### 6.2 语气

- 以码说话，文字叙述极简
- 开头直奔主题，1-2 句话定位
- 偶尔口语化："话不多说，直接上图"

### 6.3 内容组织

文章结构由主题本身决定，不套固定模板。但有几条通用原则：

- **开头快速定位**：前几行就让读者知道"这篇在讲什么、涉及哪些源码"，不要铺垫太长
- **先给全景再钻细节**：如果主题涉及多个类/函数，先用一张总览图或一段话交代整体关系，再逐个展开
- **关键函数要配调用图**：分析核心函数时，用 ASCII 树形图同时展示"它调用了谁"和"谁调用了它"，让读者既能往下追也能往上溯。示例：

```
[谁调用了 DoSomethingCore]                            // 被调用图
├─ SystemA::Initialize()
├─ SystemB::OnEvent()
└─ SystemC::Tick()

DoSomethingCore()                                     // 调用图（它调用了谁）
├─ ValidateInput()
├─ ★ CreateOrFindResource()                           // 核心调用
│   ├─ LookupCache()
│   └─ AllocateNew()
└─ NotifyDependents()
```

- **重要类型要画生命周期**：涉及核心类/对象/资源时，用 ASCII 步骤图展示从创建到销毁的完整生命周期，标注触发函数、状态变迁和持有关系（见 4.7 示例）
- **代码与解释交织**：不要"前半篇全是文字 + 后半篇全是代码"，而是一段解释接一段代码，像对话一样推进

### 6.4 深度与篇幅基线

核心目标：**让读者看完一篇后，能清楚这个系统是什么、怎么运转、为什么这样设计**。以下是参考基线，不是硬性计数：

1. **结构完整**：至少有“概述 + 核心内容 + 小结”的基本骨架；核心内容部分按子主题拆 ## 小节，数量由主题复杂度决定
2. **代码要讲透，不要堆砌**：贴的每段代码都应该有存在的理由（展示核心逻辑、关键数据结构、重要调用入口等），不为凑行数贴无关代码
3. **涉及 Shader 主题时**：必须同时覆盖 C++ 调度端（谁 dispatch、谁传参）和 .usf/.ush 执行端（shader 内部怎么消费这些参数），不能只贴一端
4. **图和表服务于理解**：如果一张调用链图能让读者秒懂整个流程，就画；如果一张表能把散落的字段/参数收拢对比，就列。不要为了“凑够 N 张图”而画没有信息量的图
5. **文末小结**：给出几条关键结论，说清“这个系统最值得记住的点是什么”

### 6.5 写作的核心原则

下面几条代替了之前的数字硬约束，目标是让模型用判断力而不是凑数：

1. **代码不裸贴**：每段代码前后都应有文字说明它在整个流程里的角色，以及为什么值得展示。代码与解释应该交织出现，而不是“前半篇文字 + 后半篇代码”
2. **结构体/参数块不裸列**：当出现大段字段定义时，按功能组解释语义、数据从哪来、被谁使用
3. **调用关系说清楚**：每个核心小节都应让读者知道“这段逻辑被谁调用、参数从哪来、结果给了谁”。用程序员自然的表达方式，禁止机械套用“上游→下游→闭环”句式
4. **大代码块要拆**：如果一段代码超过了一屏的阅读舒适度，就按步骤拆成多块，中间加解释性过渡

## 七、工作流程

整个分析流程分为 **两个阶段**，各阶段自动衔接，无需等待用户指示。

### 阶段一：生成 TodoList

当用户指定一个分析主题（如"分析 UE5 渲染管线"、"分析 CSM 级联阴影"）时：

1. **理解主题**：明确用户要分析的 UE5 模块/系统/类的范围
2. **Knot 预检索**：并行发起 git + git_iwiki 检索，了解该系统的整体架构、核心类、关键流程
3. **生成 TodoList 文件**：在工作目录下创建 `<Topic>_Analysis_TodoList.md`
4. **输出 TodoList 等待用户确认**

### 阶段二：逐项生成章节文档

用户确认 TodoList 后（如"开始吧"、"从 1.2 开始"）：

1. **确定当前项**：取用户指定的项，或按 TodoList 顺序取第一个 `- [ ]` 项
2. **Knot 深度检索**：针对该项的关键源码和概念，并行发起 git + git_iwiki 检索
3. **解析结果**：提取关键代码片段和架构信息
4. **编写文档**：按第二~六节的规范产出 Markdown 文件，保存到工作目录
5. **更新 TodoList**：将对应项标记为 `[x]` 并追加文件名
6. **动态扩展 TodoList**：写文章的过程中会不断发现新的关联主题，这些发现应该实时反馈到 TodoList 中。具体做法：
   - **边写边记**：写作过程中遇到值得深入但当前文章不展开的类/函数/子系统（如分析对象创建流程时遇到 `GUObjectAllocator`），立即记录下来
   - **判断是否新增**：满足以下任一条件的关联主题，应加入 TodoList：
     - 它有独立的实现逻辑（不是简单的 wrapper / 转发函数），值得单独分析源码
     - 当前文章多次引用它但只能一笔带过，读者看完会想"这个东西具体怎么工作的？"
     - 它是理解当前主题的前置依赖或后续延伸，缺了它知识链条断裂
   - 反过来，如果只是一个工具函数、标准容器用法、或者一句话就能解释清楚的概念，在当前文章里内联说明即可，不用单独开篇
   - **完成后更新**：每完成一篇文章后，回顾记录的关联主题，将符合条件的加入 TodoList（新增子项如 `1.1a`、`1.1b`，或追加到对应章节下）
   - **交叉补充**：如果发现某个主题与其他章节有交叉依赖但对方未覆盖，在对方章节描述中补充该分析点
   - **保持准确**：如果已完成章节的描述与实际产出不符，更新描述使其准确
   - TodoList 不是一成不变的初始规划，而是随着分析深入不断生长的知识地图
7. **自动继续下一项**：当前项完成后，自动取下一个 `- [ ]` 项继续，直到 TodoList 全部完成

**分析维度**：

每篇文章根据主题选择合适的分析角度，常见维度包括但不限于：

- **函数调用链**：从入口到核心实现的调用路径，用树形 ASCII 图展示
- **生命周期**：对象从创建到销毁的完整过程，谁创建、谁持有、谁销毁
- **主要作用**：解决什么问题，为什么需要它
- **数据格式与数据结构**：关键结构的内存布局、字段含义、数据流向
- **线程模型**：在哪个线程执行，跨线程时怎么同步
- **调用关系**：被谁调用、调用了谁，与哪些系统有协作
- **性能相关**：关键优化手段、CVar 控制开关
- 等等

模型应根据主题自行判断哪些维度最相关，不需要每篇都覆盖所有维度。但有两条硬规则：

- 涉及数据编解码的主题，必须覆盖数据格式
- 涉及跨线程 / TaskGraph 的主题，必须覆盖线程模型

**节奏要求**：

- **每篇文章至少一轮对话完成**：简单章节在一轮对话内完成检索 + 编写 + 保存的全流程
- **复杂章节允许多轮对话**：如果涉及大量源码、多个核心函数或复杂流程，可拆分为多轮完成（如第一轮检索并编写框架和主要内容，后续轮补充细节），但每轮结束时都要保存当前进度到文件
- **自动连续处理**：完成当前项后自动进入下一个 `- [ ]` 项，无需等待用户指示，直到 TodoList 全部完成

### 7.1 完成前质量自检

在将 Todo 项标记为 [x] 前，用以下几个问题自检：

1. **读者能看懂吗？** —— 每段代码都有解释，每个小节都说清了“这段逻辑被谁调用、参数从哪来、结果给了谁”
2. **有没有裸贴代码？** —— 不存在“只有代码、几乎无中文注释”的块；结构体/参数块不是裸列字段，而是按组解释了语义
3. **图和表有信息量吗？** —— 每张图/每张表都在帮助理解，而不是为了凑数
4. **Shader 主题是否两端都覆盖了？** —— C++ 调度端 + .usf 执行端，不能只贴一端
5. **是否落到了源码？** —— 不存在“复制官方文档描述但没有对应到具体源码位置”的段落
6. **有没有漏掉值得独立分析的关联主题？** —— 文中提到但未展开的关键类/函数/子系统，是否已追加到 TodoList

## 八、TodoList 规范

### 8.1 格式

TodoList 使用 Markdown checkbox 格式：

```markdown
- [x] **1.1 CSM 基本原理**
  - [x] 1.1.1 标题 → `01_01_01_FileName.md`
  - [x] 1.1.2 标题 → `01_01_02_FileName.md`
  - 关键源码: `file1.h`, `file2.cpp`
  - [ ] 补充审查：回顾本章内容，将发现的关联主题追加到 TodoList
```

### 8.2 完成标记

- 未完成：`- [ ]`
- 已完成：`- [x]` 并在行尾追加 ` → `文件名.md``
- 章节级别完成：`- [x] **X.Y 章节标题**`

### 8.3 层级结构

```
## 大部分标题
- [x] **X.Y 章节标题**
  - [x] X.Y.Z 小节标题 → `XX_YY_ZZ_FileName.md`
  - 关键源码: `file.cpp`
  - 关键概念: Concept1, Concept2
  - [ ] 补充审查：回顾本章，追加关联主题
```

### 8.4 TodoList 文件模板

```markdown
# [主题名] 源码分析 TodoList

> 生成日期: YYYY-MM-DD
> 分析范围: [简述分析边界]

---

## 第一部分：[大章节标题]

- [ ] **1.1 [章节标题]**
  - [ ] 1.1.1 [小节标题]
  - [ ] 1.1.2 [小节标题]
  - 关键源码: `File1.cpp`, `File2.h`
  - 关键概念: Concept1, Concept2
  - [ ] 补充审查：回顾本章内容，将发现的关联主题追加到 TodoList

- [ ] **1.2 [章节标题]**
  - [ ] 1.2.1 [小节标题]
  - 关键源码: `File3.cpp`
  - [ ] 补充审查：回顾本章内容，将发现的关联主题追加到 TodoList

## 第二部分：[大章节标题]
...

## 附录（可选）

- [ ] A.1 源码文件索引表
- [ ] B.1 完整数据流图
- [ ] C.1 CVar 速查表
```

## 九、文件命名规范

| 类型     | 格式                                   | 示例                               |
| -------- | -------------------------------------- | ---------------------------------- |
| TodoList | `<Topic>_Analysis_TodoList.md`         | `CSM_Analysis_TodoList.md`         |
| 分析文档 | `XX_YY_ZZ_<EnglishTitle>.md`           | `01_01_01_System_Overview.md`      |
| 附录     | `Appendix_<Letter><Number>_<Title>.md` | `Appendix_A1_Source_File_Index.md` |

- `XX` — 大部分编号（两位数字）
- `YY` — 章节编号（两位数字）
- `ZZ` — 小节编号（两位数字）
- `<EnglishTitle>` — 英文标题，单词间下划线连接，首字母大写
